## Что вызвало churn и переработку

### 1. Регрессия переводов Story 3.2 → Story 3.4
**Факт:** Story 3.2 добавила `routing.official.*` и `routing.business.*` EN-переводы в `lang-toggle.js`, но **пропустила UK-переводы**. Это не было поймано в review Story 3.2. Story 3.4 наткнулась на failing tests в `business-next-step.test.js` и была вынуждена исправлять чужую регрессию.

**Корневая причина:** нет cross-story проверки translation completeness (EN + UK должны быть в паре). Тест существовал, но запускался только в контексте Story 3.4.

### 2. Selector drift в E2E тестах Story 3.2
**Факт:** Story 3.2 добавила второй CTA → Дія (routing card), сломав тест из Story 3.1 (`official-check.spec.ts`) — он ожидал один CTA, нашёл два. Пришлось scope-ировать selector к `.info-page__cta-section`.

**Корневая причина:** добавление нового контента на существующую страницу без проверки существующих E2E-тестов на этой странице.

### 3. Timing-зависимость в fallback detection (Story 3.3)
**Факт:** fallback в `request-form.js` использует `visibilitychange` event + 1.8s timeout. Это timing-sensitive — E2E тест в Story 3.4 потребовал явного `wait timeout` и правильного `form submission trigger` для стабильной работы.

**Корневая причина:** архитектура fallback detection через `visibilitychange` inherently ненадёжна для автоматизированного тестирования. Это известное ограничение `mailto:` fallback detection.

### 4. Route naming: `/perevir-product/` vs контента
**Факт:** Story 3.1 создала страницу по пути `/perevir-product/`, хотя контент — "official checking guidance". Несовпадение было помечено в story record как `[AI-Review][Should-Fix]` с аннотацией "log decision or plan rename for later". Принято решение оставить как есть в Phase 1.

**Статус:** технический долг, зафиксированный и осознанный, не блокирует.

### 5. `lang-toggle.js` как узкое место
**Факт:** Все 4 истории Epic 3 модифицировали `lang-toggle.js`. Кумулятивный рост: +11+11 (3.1) + +10+10 (3.2) + +62 (3.3) + UK-fix (3.4). Это создаёт риск key collision и сложность проверки полноты переводов.

## Тестирование и code review: уроки

### 1. Правило парных переводов не проверяется автоматически
Ни один существующий тест не проверяет, что для каждого EN-ключа есть соответствующий UK-ключ. Story 3.2 добавила 10 EN-ключей без UK — и это прошло все тесты до Story 3.4.

**Действие для Epic 4:** при добавлении переводов в `lang-toggle.js` — явная проверка: "каждый новый `en.*` ключ имеет пару `uk.*`". Включить в definition of done на уровне story checklist.

### 2. E2E тесты на общих страницах = совместная собственность
Когда Story 3.2 изменила `/perevir-product/`, тест из Story 3.1 (`official-check.spec.ts`) сломался. Стандартный `npm run test:smoke` перед Story 3.2 commit выявил бы это раньше.

**Действие для Epic 4:** запуск полного smoke suite обязателен перед описанием истории как `done`. Не только тесты новой истории — весь smoke.

### 3. Timing-sensitive E2E тесты требуют явных wait-стратегий
Story 3.4 исправила E2E тест, добавив правильный wait + submission trigger для fallback. Это должно было быть в Story 3.3 изначально.

**Действие для Epic 4:** E2E тесты для async/timeout-зависимого поведения обязательно включают явные `await page.waitFor*()` вместо poll-based или time-based ожиданий.

### 4. `[AI-Review][Should-Fix]` нотации — ценный паттерн
Story 3.1 использовала явные review-нотации в story record с указанием файла и описанием проблемы. Все были resolved до закрытия истории. **Этот паттерн стоит сохранить в Epic 4.**

### 5. Разные модели = разные стили реализации
3.1 (opus-4-6), 3.2 (sonnet-4-6), 3.3 (nemotron-super), 3.4 (sonnet-4-6). Story 3.3 — самая крупная (214-строчный HTML, 237-строчный JS, 885-строчные тесты). Несмотря на разные модели, архитектурный контракт выдержали все.
