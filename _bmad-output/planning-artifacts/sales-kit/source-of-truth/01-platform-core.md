# genu.mark Core Model

## Main object of sale

For regulated `еАкциз` motion, the thing being sold is `genu.mark`.

## What is core

`genu.mark` is the line core.

It covers:
- receipt of codes as input;
- code application;
- verification;
- reject / rework control;
- aggregation;
- packaging result.

## What is external

### ERP

`ERP` is external in two distinct roles:
- `ERP -> source of production job`;
- `ERP / internal systems <- exchange of results and statuses`.

Any regulated source or output that says only `ERP integration` without this split is incomplete.

### еАкциз

`еАкциз` is a separate external state contour.

It connects to the solution, but it is not the product core.

### Other internal systems

Other customer systems may connect where needed, but they do not redefine the core.

## Role split

- `Dominanta` = Domino equipment and commercial entry point;
- `Logictime` = architecture, integrations, exceptions, implementation, launch;
- `genu.im` = supporting proof context only when needed;
- `genu.code` = separate voluntary branch.

## Mandatory rendering rule

Every regulated source and output must make four things visible:
1. `genu.mark` is the thing being bought;
2. the core sits inside `genu.mark`;
3. `ERP` is external in two roles;
4. `еАкциз` is a separate external contour.

## Commercial implication

The correct explanation order is:
1. implementation is inevitable;
2. the buyer is choosing implementation quality;
3. `genu.mark` is the strong core;
4. external contours connect deliberately;
5. strong core first reduces launch risk;
6. next step = audit / architecture session / pilot.

## Do not imply

- platform before product;
- `genu.im` before `genu.mark`;
- `genu.code` in early regulated narrative;
- `ERP` or `еАкциз` as part of the core;
- public proof as legal approval.
