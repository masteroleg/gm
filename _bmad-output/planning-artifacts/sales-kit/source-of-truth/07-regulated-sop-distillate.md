# Regulated SOP Distillate

## Purpose

This file extracts only the reusable truths from the raw `genu.mark` `еАкциз` SOP.

It exists to strengthen:
- technical presale materials;
- evidence and audit-ready materials;
- edge-case and exception materials;
- matching QA logic.

It must not turn buyer-facing assets into a procedure manual.

## Use boundary

Use this file for:
- compact upstream reinforcement;
- technical and evidence credibility;
- exception and responsibility language.

Do not use it to:
- replace the core control packet;
- replace the buyer-facing source assets;
- upload the raw SOP into the default regulated NotebookLM pack;
- overload executive materials with legal or operator terminology.

## 1. What the SOP confirms about the core

The SOP supports the already locked architecture:

- `genu.mark` remains the line core;
- the core covers code intake, application, verification, reject / rework control, aggregation, and line result state;
- external systems do not redefine the core;
- strong launch discipline depends on keeping this boundary explicit.

## 2. What the SOP confirms about external contours

The SOP reinforces that the outer contours must stay visible and separate:

- `ERP -> source of production job`;
- `ERP / internal systems <- exchange of results and statuses`;
- `еАкциз` as a separate external state contour through cabinet, API, or offline exchange;
- legal and reporting actions happen at the boundary, not inside the core itself.

This strengthens the rule that vague `ERP integration` wording is insufficient.

## 3. Distilled exception model

The SOP gives concrete support for a stronger exception story.

### Verification and reject / rework

- unreadable, duplicated, or invalid codes must be detected;
- failed units must move to a reject branch, not disappear in manual ambiguity;
- rework and relabel scenarios need controlled data continuity;
- bad physical flow must not silently corrupt digital state.

### Aggregation breaks

- incomplete cases;
- broken pack relationships;
- repacking;
- de-aggregation and re-aggregation;
- regeneration of higher-level identifiers when pack composition changes.

### Offline and recovery

- offline operation exists as a bounded mode, not as uncontrolled improvisation;
- offline periods require later synchronization;
- recovery after line stop, connectivity loss, or restart must preserve traceability;
- launch readiness depends on recovery logic being designed before go-live.

### Manual intervention

- manual touchpoints still exist in real production;
- the requirement is not to pretend they do not exist;
- the requirement is to make them visible, reviewable, and controlled.

## 4. Distilled evidence model

The SOP is especially strong as proof that evidence must be designed in from the start.

### Evidence on the unit

- what code was used;
- when it was applied;
- whether verification passed or failed;
- whether the unit was rejected or accepted.

### Evidence on packaging hierarchy

- unit -> case -> pallet relationships;
- group and transport identifiers;
- closure, reopening, repacking, and hierarchy changes;
- proof that aggregation structure remained reviewable.

### Evidence on state transitions

- activation state;
- warehouse / in-transit / retail state changes where relevant;
- shipment and acceptance events;
- exception states that block release until resolved.

### Evidence on user and system actions

- user actions;
- signing or approval events where applicable;
- generated reports, logs, and trace records;
- visible history for review, incident analysis, and audit preparation.

## 5. Distilled responsibility model

The SOP strengthens the idea that safer rollout depends on visible responsibility boundaries.

Buyer-facing language should not copy every legal term, but it should preserve these truths:

- line behavior and control logic must be owned explicitly;
- external-system exchange must be owned explicitly;
- state exchange with `еАкциз` must be owned explicitly;
- user approval, sign-off, and logs must not be hand-waved;
- review checkpoints must exist before release, shipment, and rollout expansion.

## 6. Buyer-safe language that the SOP can support

These are safe downstream ideas to reuse after adaptation:

- `evidence is left by design, not added later`;
- `reject, rework, and aggregation logic are part of the real launch contour`;
- `manual risk is reduced by making events and states visible`;
- `offline and recovery logic should be designed before launch`;
- `a safer rollout depends on explicit boundaries and reviewable state changes`;
- `strong core first makes external handoffs easier to govern`.

## 7. Language to avoid copying from the raw SOP

Do not copy raw SOP style into buyer-facing materials:

- full procedure chains;
- dense legal abbreviations without audience need;
- article-style clause numbering;
- operator instructions;
- language that sounds like a compliance manual;
- wording that could be read as a legal guarantee.

Translate raw SOP depth into:

- calm technical credibility;
- visible exception discipline;
- reviewable evidence;
- clearer split of responsibility.
