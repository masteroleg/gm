# Final Regulated Control Packet

## Mandatory use

Use this file first for every mandatory regulated NotebookLM run.

## What is sold

- Sell `genu.mark`.
- Sell safe regulated implementation around the line core.
- Sell `Dominanta + Logictime` as the safer path to rollout.

## Core formula

Render `genu.mark` everywhere as the simple strong line core that:
- receives codes;
- applies;
- verifies;
- aggregates;
- gives packaging result.

## Non-negotiable narrative order

1. `еАкциз` implementation is inevitable.
2. The buyer's question is how to launch without production risk.
3. `genu.mark` is the line core.
4. `ERP` and `еАкциз` are external contours.
5. Strong core first reduces launch risk.
6. `Dominanta + Logictime` is the safer implementation path.
7. CTA = audit / architecture session / pilot.

## Non-negotiable architecture rendering

- `genu.mark` = core.
- `ERP -> source of production job`.
- `ERP / internal systems <- exchange of results and statuses`.
- `еАкциз` = separate external state contour.
- Never collapse the two ERP roles into vague `ERP integration` wording.

## Value order

Use this order unless a later supporting proof section is explicitly needed:
1. `genu.mark`
2. Domino / `Dominanta`
3. `Logictime` implementation discipline
4. `genu.im` only as supporting proof

## Output rules

- Output language = modern, buyer-facing Ukrainian.
- Tone = calm, industrial, credible, low-drama.
- Final output must be forwardable without oral explanation.
- CTA must point to audit / architecture session / pilot.
- If a thought can be explained through `genu.mark`, do not explain it through `genu.im`.
- `genu.im` may appear only as supporting proof context, never as the opening frame.

## Instant fail gates

FAIL immediately if any output:
- sells a platform instead of `genu.mark`;
- shows `ERP` as part of the core;
- shows `еАкциз` as part of the core;
- mentions `genu.code` in the early regulated narrative;
- mixes regulated and voluntary logic;
- sounds like an operator draft or internal note;
- omits a next step to audit / architecture session / pilot.
- introduces any new product, architecture layer, product branch, or platform explanation not present in the approved source packet;
- uses unsupported broad superiority claims or weak claims such as `best equipment`, `turnkey`, `single responsibility`, or `full package`.
