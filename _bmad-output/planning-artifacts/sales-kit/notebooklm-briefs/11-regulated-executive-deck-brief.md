# Executive Deck Brief / Regulated

## Purpose

Generate the full regulated executive deck with the final approved narrative order.

## Deliverable

- deliverable type: 10-12 slide executive deck by default
- audience: CEO / owner / GM / sponsor
- output language: modern buyer-facing Ukrainian
- required CTA: audit / architecture session / pilot
- target range: 10-12 slides
- minimum range: 8-9 slides only if meaning is not lost
- maximum range: up to 14 slides only if extra slides close distinct mandatory buying-questions
- do not optimize for slide count itself; optimize for clarity of selling `genu.mark`

## Required sources

Upload only:
- `regulated-final-bundle/01-control-packet.md`
- `regulated-final-bundle/02-approved-regulated-source-packet.md`
- `buyer-facing/regulated/01-executive-deck-regulated.md`
- `notebooklm-briefs/11-regulated-executive-deck-brief.md`

## Non-negotiables

- sell `genu.mark`, not an abstract platform;
- keep `genu.mark` as core;
- make the first screen and first paragraph immediately show that `genu.mark` is the thing being sold;
- render `genu.mark` as the simple strong core that receives codes, applies, verifies, aggregates, and gives packaging result;
- show the exact wording `ERP -> source of production job`;
- show the exact wording `ERP / internal systems <- exchange of results and statuses`;
- keep `еАкциз` external;
- do not mention `genu.code` in the early regulated narrative;
- do not mix voluntary logic;
- do not introduce any new product, architecture layer, platform explanation, or selling object not present in the approved sources;
- do not use vague wording like only `ERP integration`;
- do not use claims like `найкраще`, `№1`, `під ключ`, `гарантує`, `повністю закриває`, `гарантований комплаєнс`, `єдина відповідальність`, `повний пакет`;
- end with audit / architecture session / pilot.

## Required structure

Use this order as the mandatory executive spine:

1. inevitable `еАкциз` launch and the buyer question about safe implementation
2. `genu.mark` as the line core
3. simple core formula: receives codes, applies, verifies, aggregates, gives packaging result
4. `ERP` and `еАкциз` as external contours
5. strong core lowers rollout risk
6. interface and exception risk lives on the handoffs
7. `Dominanta + Logictime` as safer path
8. operational outcome
9. next step CTA

You may compress to 8-9 slides only if no mandatory meaning is lost.
You may expand to 13-14 slides only if each additional slide clearly closes a distinct buying-question.

If the deck uses 13-14 slides, explicitly state which buying-question each extra slide closes:
- what is being sold;
- what is core;
- what is external;
- why this path is safer;
- what the next step is.

Architecture slide must contain both exact ERP lines:
- `ERP -> source of production job`
- `ERP / internal systems <- exchange of results and statuses`

Interface / exception section must make handoff and exception risk concrete, not generic:
- interface failures;
- offline / recovery;
- reject / rework;
- aggregation breaks or packaging exceptions.

## Editorial standard

- calm and industrial, not theatrical;
- concise slide titles;
- no draft notes, bridge lines, or operator wording;
- no inflated superiority language;
- no `єдина відповідальність за результат`, `найкраще обладнання`, or equivalent overclaims;
- the buyer should understand the deck without a presenter filling gaps.
- after reading, the buyer should be able to retell what is sold, what is core, what is external, why this is safer, and what the next step is.
