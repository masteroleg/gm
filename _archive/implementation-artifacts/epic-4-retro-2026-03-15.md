## Что вызвало churn и переработку

### 1. Baseline regression при старте 4.1 — скрытый долг Epic 3
Story 4.1 при baseline smoke обнаружила, что `[data-routing-official]` карточка в `/perevir-product/` была тихо удалена code review hotfix-ом из Epic 3 (commit `6597406` — "fix: remove duplicate Diia CTA"). Тот fix неверно классифицировал карточку как duplicate, тогда как она была deliverable из Story 3.2.

**Корневая причина:** code review fix в одном эпике не проходит full smoke. Hotfix прошёл только targeted проверку. Паттерн "code review fix → silent regression" — новый класс риска.

**Исправлено:** hotfix до merge 4.1 (commit `da64873`, `ce13f08`). Story 4.1 чистая.

**Для Phase 2:** full smoke после каждого hotfix/code-review-fix, который трогает existing site content.

### 2. Story 4.3 — 4 should-fix из-за модели
Dev agent в 4.3 — `haiku-4-5`. Produced документация содержала фактически неверные утверждения о GSC. Code reviewer (sonnet-4-6) исправил все четыре до approve.

**Паттерн:** haiku-4-5 справляется с механическими изменениями (4.1: 0 should-fix), но documentation stories с non-obvious measurement concepts требуют модели уровня sonnet+.

**Для Phase 2:** model selection policy — документировать в project-context.md.

### 3. Status sync — 3-й эпик подряд
`epic-4` не обновился до `done` после завершения последней истории. Это повторилось в Epic 2, Epic 3, Epic 4. Не поведенческая проблема — структурная. Требует enforcement mechanism, не reminder.

## Shared-touch и hidden dependency риски для Phase 2

### Паттерн "code review fix → silent regression" (новый, впервые чётко проявился в Epic 4)

**Механизм:**
1. Dev или reviewer в эпике N делает "очевидный" fix
2. Fix удаляет что-то, что кажется лишним
3. Но это deliverable из эпика N-1
4. Smoke тест не охватывает scope того review fix-а

**Митигация:** full smoke run после каждого hotfix/code-review-fix, не только после feature commit.

### `request-form.js` — аккумулятор зависимостей

Файл модифицировался в Stories 3.3, 3.4, 4.2. В Phase 2 это горячая зона. Нужна явная owned API + integration test для всего экспортируемого интерфейса.

### Тестовое покрытие (`docs/`) — слепая зона

Unit tests и smoke tests не могут проверить фактическую точность measurement documentation. Только code review. Для story типа "создать review guide" — code review обязателен и с достаточно сильной моделью.

## Итоговые уроки Epic 4

1. **Separation contract между историями работает.** 4.1 → 4.2 → 4.3 — три чётких слоя без смешения. Это проектировать заранее, не постфактум.

2. **Forbidden list в Dev Notes = guardrail, не пожелание.** Explicit "what this story must NOT do" предотвратил scope creep при реализации.

3. **Code review для docs — не бонус, а необходимость.** Factual accuracy в measurement guides нельзя протестировать автоматически.

4. **Model selection влияет на quality profile.** haiku-4-5 быстрее для механических задач. sonnet-4-6 надёжнее для задач с accuracy requirements. Это не проблема модели — это проблема выбора.

5. **Status sync — structural, не behavioral.** Три consecutive epic failures — это дизайн процесса, не забывчивость.

6. **Baseline smoke перед любым change — неотменяемо.** Story 4.1 поймала Epic 3 regression именно потому что выполнила baseline smoke до написания кода.
