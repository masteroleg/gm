# genu.im Homepage Master Plan

Last updated: 2026-03-09

## Key decisions

- Use `–µ–Р–Ї—Ж–Є–Ј` and `–Ф—Ц—П` in Ukrainian at all times.
- Treat `–µ–Р–Ї—Ж–Є–Ј` / marking as the strategic core.
- Catch two homepage audiences early:
  - `–µ–Р–Ї—Ж–Є–Ј`
  - responsible manufacturers
- Separate those branches later in the page instead of mixing them from the start.
- Keep `verification preview` mandatory, but introduce it after category framing and before deeper branch detail.
- Add trust placeholders now; fill them in during later epics.

## Priority ladder

### P0 - Remove trust damage

1. Fix false bilingualism.
2. Fix `–µ–Р–Ї—Ж–Є–Ј` / `–Ф—Ц—П` / browser-flow ambiguity.
3. Remove or soften unsupported claims.
4. Remove internal-looking all-caps labels.
5. Normalize terminology, tone, and register.

### P1 - Make the homepage understandable

1. Rewrite hero to catch both categories.
2. Rebuild proof strip with only contextualized claims.
3. Replace vague CTAs with explicit next-step CTAs.
4. Expand header/navigation.
5. Add baseline trust floor and placeholder destinations.

### P2 - Fix reveal sequence and product explanation

1. Add bridge section: not just a QR.
2. Place the mandatory verification preview after that bridge.
3. Build the preview around real result states and fields.
4. Split the page into `–µ–Р–Ї—Ж–Є–Ј` and responsible-manufacturer branches.

### P3 - Build evidence and conversion support

1. Replace `Why genu.im` with compare-vs-generic-QR.
2. Rebuild `How it works` as an operational flow.
3. Rebuild `Proof at scale` as an evidence layer.
4. Add FAQ / objection handling.
5. Add NDA-safe proof framing.

### P4 - Validate and iterate

1. 5-second clarity check.
2. Regulatory wording review.
3. CTA comprehension review.
4. Mobile hierarchy review.
5. Final language consistency pass.

## Staged execution plan

### Stage 0 - Language and terminology discipline

Goal: remove mixed-language and regulatory precision issues before structural work.

Tasks:
- keep `–µ–Р–Ї—Ж–Є–Ј` and `–Ф—Ц—П` in Ukrainian
- remove `eAktsyz` / `Diia`
- remove false bilingual UI fragments
- align terminology across hero, chips, preview, CTA, footer

Result:
- page stops looking inconsistent and careless

### Stage 1 - Stop-loss trust cleanup

Goal: remove the elements that actively hurt trust.

Tasks:
- cut or soften risky claims
- remove internal-looking labels
- reduce proof strip to supported points
- swap vague CTAs for explicit actions
- add trust footer placeholders

Result:
- page stops undermining itself

### Stage 2 - Catch both categories early

Goal: make the first screen clear and broad enough to catch both target audiences.

Tasks:
- rewrite hero
- add category framing
- add early dual-context cues for `–µ–Р–Ї—Ж–Є–Ј` and responsible manufacturers
- expand top navigation for trust discovery

Result:
- visitors understand what the product is and who it is for

### Stage 3 - Introduce the mechanism in the right order

Goal: explain why this is not just another QR flow.

Tasks:
- add bridge section before preview
- explain code vs checkable record
- place verification preview after the bridge

Result:
- the user first understands the category, then the mechanism

### Stage 4 - Make verification preview real proof

Goal: turn preview into a product truth artifact.

Tasks:
- show `valid` / `suspicious` / `no record`
- include `record ID`, `status`, `source`, `timestamp`, `product reference`, `next action`
- connect preview to the narrative

Result:
- proof lives in the interface, not in slogans

### Stage 5 - Split the two branches cleanly

Goal: separate two stories without losing the shared top-level message.

Tasks:
- create `–µ–Р–Ї—Ж–Є–Ј` branch
- create responsible-manufacturer branch
- give each branch different emphasis and CTA routing

Result:
- messages stop competing on the same screen

### Stage 6 - Rebuild weak sections

Goal: replace repetitive or weak sections with stronger explanatory ones.

Tasks:
- replace `Why genu.im`
- rebuild `How it works`
- rebuild `Proof at scale`
- add FAQ
- define NDA-safe evidence format

Result:
- homepage explains and proves instead of repeating itself

### Stage 7 - Add proper placeholders

Goal: create a clean trust surface before full content is ready.

Tasks:
- add placeholder destinations for `About`, `Contact`, `Proof & Cases`, `Privacy`, `Terms`, `FAQ`
- add placeholder CTA destinations

Result:
- trust surface exists now and can be filled later

### Stage 8 - Validate

Goal: confirm the new structure works.

Tasks:
- clarity review
- regulatory review
- CTA review
- mobile review
- language review

Result:
- changes are clearer, safer, and more convincing

Create a homepage v2 blueprint that maps:

- draft headline/subhead/CTA per section
